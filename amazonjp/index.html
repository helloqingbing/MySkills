Usage guide / Amazon Japan launch system

从选品到复盘，这套流程怎么跑

这不是一张只用来看的流程图，而是一套可执行的工作流。每个候选品都用同一组 ID 串联数据、证据、Listing、广告和复盘；每个阶段必须留下产出物和 Gate 状态，才能进入下一步。

最小可执行闭环
12阶段：实时数据、选品、QC、门禁、上架、FBA、投放、复盘
14skills：覆盖研究、供应链、Seller Central、FBA 和经营复盘
3Gate 结果：Pass、Needs Proof、Blocked
1总控：amazon-jp-sop-orchestrator 维护状态机和交接包

四条运行规则

只要遵守这四条，流程就不会变成散乱的“分析报告堆”。每个阶段都能被追踪、复用和复盘。

ID

一品一条主线

从第一张表开始保留 candidate_id，成 SKU 后继续补 sku、parent_asin、launch_id。

Gate

没有门禁不推进

任何阶段都必须给出 pass、needs_proof 或 blocked，特别是 Listing、FBA、广告前。

Proof

证据先于结论

合规、功效宣称、认证、成本、毛利、样品 QC 都要进入 evidence_matrix.csv。

Loop

投放数据回流

搜索词、ACOS、TACOS、评论、退货原因要回到关键词、Listing、A+、产品规格和广告结构。

Skills 解决的问题

这 14 个 skills 不是独立报告模板，而是一组可串联的岗位能力。每个 skill 负责一个明确问题，并把产出交给下一阶段。

总控

amazon-jp-sop-orchestrator

解决流程散、文件乱、阶段无法追踪的问题。

用法：
先调用它创建 launch workspace、状态机、Gate 和交接包。
产出：
candidate_id、sku、artifact contract、实时数据与周度复盘入口。
实时

amazon-jp-live-data-sync

解决运营数据滞后、报表字段分散、广告和库存无法一起复盘的问题。

用法：
导入 Business Report、Search Term、库存、退货和评论导出。
产出：
live_data/*.normalized.csv、数据质量说明和告警输入。
数据

amazon-data-ingestor

解决 POE、SQP、竞品、供应商报价、广告报表字段不统一的问题。

用法：
把原始 CSV 或粘贴表格先交给它清洗。
产出：
normalized/*.csv，保留证据字段和可关联 ID。
评分

amazon-opportunity-scorer

解决“哪个品值得继续验证”的排序问题。

用法：
输入清洗后的需求、增长、竞争、价格和风险数据。
产出：
Top N 机会、A/B/C/Reject、评分依据和淘汰理由。
风控

amazon-risk-auditor

解决早期选品忽略 IP、合规、FBA 限制、品牌垄断和季节性的风险。

用法：
在推荐候选品前先做风险筛查。
产出：
pass、needs_proof、blocked_pending 和风险证据。
报告

amazon-radar-report

解决机会清单不能转成行动计划的问题。

用法：
在评分和风险筛查后生成周报或月报。
产出：
机会雷达、差异化方向、风险摘要和 7 日验证计划。
供应链

amazon-jp-sourcing-qc-planner

解决供应商、样品、包装和 QC 没有形成可验收证据的问题。

用法：
输入供应商报价和差评痛点，生成样品评估和验货标准。
产出：
supplier_scorecard.csv、qc_checklist.csv、sample review。
硬门禁

amazon-jp-launch-gate

解决 Listing、FBA、广告上线前缺少 SKU 级放行判断的问题。

用法：
在上架前检查类目、JAN/GTIN、IP、标签认证、进口/FBA、毛利和 claim proof。
产出：
pass、needs_proof 或 blocked 的 launch gate report。
产品

amazon-review-miner

解决差评只被总结、没有转成产品改进和 QC 要求的问题。

用法：
输入竞品评论，聚类 1-3 星痛点和可修复缺陷。
产出：
product_requirements.csv、供应商要求、A+ 反驳点和 claim caution。
内容

amazon-jp-listing-aplus-designer

解决日本站 Listing、A+、图片 brief、关键词主表分散产出的问题。

用法：
在 Launch Gate 不阻塞后，生成日文标题、五点、描述、A+ 模块和图片说明。
产出：
Listing package、A+ module map、amazon_keyword_master.csv。
后台

amazon-jp-seller-central-ops

解决 Listing 包到 Seller Central 实操之间缺少字段、flat file 和人工证明的问题。

用法：
在 Listing/A+ 完成后生成后台上传清单和操作日志。
产出：
seller_central.checklist.csv、seller_central.ops_log.csv。
FBA

amazon-jp-fba-shipment-planner

解决首批备货、标签、箱规、入仓和补货风险靠人工经验的问题。

用法：
输入库存、日销、包规、生产和运输周期，计算发货与补货动作。
产出：
fba.shipment_plan.csv、补货告警和执行 checklist。
广告

amazon-ads-planner

解决广告结构、预算、CPC、否词和复盘规则没有统一口径的问题。

用法：
输入 SKU 经济模型、关键词主表、竞品 ASIN 和搜索词报告。
产出：
campaigns.csv、negatives.csv、30 天优化计划和搜索词回流动作。
复盘

amazon-jp-performance-reviewer

解决上线后只看广告、不联动库存、利润、评论和退货的问题。

用法：
按 7/14/30 天或周度导入实时数据，输出经营决策。
产出：
performance.actions.csv、告警、scale / optimize / pause / kill。

可直接复制的 ChatGPT 指令

在 ChatGPT/Codex 里按阶段粘贴，替换 SKU、品类、数据文件或表格。每次都要求输出 Gate 结果和缺失证据。

创建项目工作区

使用 amazon-jp-sop-orchestrator，为候选品 [产品名] 创建 Amazon JP launch workspace。请生成阶段目录、artifact contract、ID 规则，并列出从 raw_data 到 optimizing 的状态机。

导入实时运营数据

使用 amazon-jp-live-data-sync，解析我提供的 Business Report、Search Term Report、库存、退货和评论导出。请统一字段、保留 launch_id / sku / asin，并输出数据质量问题和需要立刻处理的告警。

选品评分与短名单

使用 amazon-data-ingestor 和 amazon-opportunity-scorer，解析我提供的 POE/SQP/竞品/供应商数据。请按 Amazon Japan 跨境电商标准输出 Top N 机会、评分依据、淘汰理由和下一步验证任务。

供应商与 QC

使用 amazon-jp-sourcing-qc-planner，把供应商报价、样品结果和竞品差评痛点转成 supplier_scorecard、qc_checklist、包装/标签要求和 launch gate 可用的证据清单。

上架前硬门禁

使用 amazon-jp-launch-gate，检查 SKU [SKU] 是否可以进入 Listing/FBA/广告。请基于 evidence_matrix、landed_cost_model、claim ledger 输出 pass / needs_proof / blocked，并列出阻塞项。

Listing、A+ 与广告

使用 amazon-review-miner、amazon-jp-listing-aplus-designer、amazon-ads-planner，为 SKU [SKU] 生成日文 Listing、A+ 模块图文 brief、amazon_keyword_master、SP 广告结构和 30 天优化计划。

后台与 FBA 执行

使用 amazon-jp-seller-central-ops 和 amazon-jp-fba-shipment-planner，把 Listing 包转成 Seller Central 上传字段、人工操作日志、FBA 标签/箱规 checklist、首批发货数量和补货告警。

7/14/30 天复盘

使用 amazon-jp-performance-reviewer，基于 weekly metrics、广告搜索词、库存、评论和退货数据，为 SKU [SKU] 输出 scale / optimize / pause / kill 决策，并列出广告、Listing、FBA、QC 的具体动作。

每个 SKU 的交付物包

一个 SKU 能不能被团队接手，取决于这些文件是否齐全。缺一个，后续就会靠口头记忆，复盘也会断。

研究包

normalized/*.csv、opportunities.csv、竞品 ASIN、关键词种子、风险审计和 radar shortlist。

供应链包

supplier_scorecard.csv、qc_checklist.csv、sample review、包装要求、标签证据和供应商整改动作。

上市包

evidence_matrix.csv、landed_cost_model.csv、claim ledger、产品需求、Listing、A+、图片 brief。

执行包

seller_central.checklist.csv、后台操作日志、fba.shipment_plan.csv、标签/箱规 checklist 和补货告警。

增长包

amazon_keyword_master.csv、广告 campaign 文件、否词、搜索词反馈、周度指标、告警和下一轮动作。

下一步判断

每次复盘只做三类决策，避免会议变成泛泛讨论。

Pass

证据齐、毛利达标、风险可控。进入下一阶段，更新状态机，并保留当前版本产物。

Needs Proof

方向可继续，但必须补证据。给负责人、截止时间和所需文件，不允许直接进入广告或 FBA。

Blocked

类目、IP、合规、毛利或执行条件不可接受。归档原因，可作为反例沉淀到选品规则。
