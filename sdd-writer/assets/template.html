AI 辅助工程实践

5.1四角色分工

项目在 ~/.claude/CLAUDE.md 与项目根 CLAUDE.md 中把 AI 协作机制制度化：不同 AI 定位为明确的工程角色，skill 只引用抽象角色名，用户可以通过改映射替换底层提供商。

designer · Claude
主架构师。负责设计方案与 Plan 编写，产出 [PLAN REVIEW REQUEST] 报 reviewer 评审。
executor · Claude
代码实现者。遵循 Surgical Changes 原则：每行改动可追溯到用户需求，不做顺手重构。
reviewer · Codex
独立质量门禁。基于 Rubric 打分，overall ≥ 7.0 且单项无 ≤ 3 才通过；最多 3 轮。
inspiration · Gemini
创意来源。显式标记为"不可盲从"，只提供候选，决策权在 designer。

5.2双检查点 Peer Review

检查点触发时机交付格式通过阈值
Plan Review方案定稿，写码之前[PLAN REVIEW REQUEST] + --- PLAN START/END ---overall ≥ 7.0 且单项无 ≤ 3
Code Review代码完成，汇报之前[CODE REVIEW REQUEST] + --- CHANGES START/END ---同上

双检查点在代码从构思到落盘的两端各设一道闸，Rubric 打分避免"评审变鼓励"。失败回路：3 轮不过 → 升级人类决策。

5.3Async Guardrail 协作协议

Async Guardrail (MANDATORY) 规定：当 ask 返回 [CCB_ASYNC_SUBMITTED 时，主 AI 必须立即结束回合——不得轮询、不得 sleep、不得 pend。这把幂等性从工程原则提到了 AI 协作协议的层面，避免重复提交与资源浪费。

5.4四条工程原则（AI 内嵌）

Think Before Coding
显式声明假设；多种解读时摊开选项；能简化时主动说；不确定时停下发问。
Simplicity First
最少代码解决问题；拒绝推测性抽象、不可能场景的 error handling；200 行能压 50 行就压。
Surgical Changes
只动该动的；孤儿清理只针对自己造成的；禁止顺手改不相关代码或格式。
Goal-Driven Execution
任务转化为可验证的成功标准，"修 bug" → "先写能复现的测试，再让它通过"。
